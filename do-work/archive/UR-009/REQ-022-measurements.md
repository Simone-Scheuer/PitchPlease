# Standardized Measurement Output

## What
Create `js/core/measurements.js` — defines the standardized format for exercise results that feeds the skill model, practice history, session summaries, and journal.

## Detailed Requirements
- Measurement format: `{ exerciseId, exerciseType, timestamp, duration, metrics: {}, perNote: [], skills: {} }`
- Metric types: 'cents-avg', 'cents-variance', 'notes-hit-pct', 'hold-steady-ms', 'reaction-ms', 'notes-per-minute', 'time-per-note-avg'
- Per-note data: `[{ note, octave, cents, hitTimeMs, heldMs, score }]`
- Skill delta computation: maps measurements to skill dimension changes
  - pitchAccuracy: derived from cents-avg
  - pitchStability: derived from cents-variance and hold-steady-ms
  - scaleFluency: derived from notes-hit-pct and time-per-note-avg
  - reactionSpeed: derived from reaction-ms
  - range: derived from highest/lowest notes hit accurately
- Helper: `createMeasurement(exerciseConfig, evaluatorOutput)` — standardizes evaluator output into the common format
- Helper: `computeSkillDeltas(measurement)` — computes per-dimension changes
- Pure data module — just format definitions and computation, no persistence

## Context
This is Phase 1.4 of IMPLEMENTATION_PLAN.md. The measurement format is the contract between the exercise layer and the profile/history layer. Getting this right now means the profile system (Phase 7) can read data from any exercise type without special-casing.

## Builder Guidance
- Certainty level: Mostly firm — the format is defined, but skill delta computation will likely need tuning once real data flows through
- Keep the skill delta formulas simple for now — they can be refined when the profile system is built
- Ear training metrics (phrase-match, interval-accuracy) will be added later when those evaluators exist — leave extensibility but don't build stubs
