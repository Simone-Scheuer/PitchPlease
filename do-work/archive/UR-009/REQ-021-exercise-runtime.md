# Exercise Runtime Engine

## What
Create `js/core/exercise-runtime.js` — the central engine that interprets exercise configs, connects to the pitch event stream, runs evaluators, drives renderers, and manages exercise lifecycle.

## Detailed Requirements
- Factory: `create(exerciseConfig, evaluator, renderer)` returns a runtime instance
- Lifecycle: `start()`, `pause()`, `resume()`, `stop()` → returns measurements
- `onComplete(callback)` — called when exercise finishes (duration elapsed or all notes complete)
- Subscribes to `pitch` and `silence` events from the event bus
- Calls `evaluator.onPitch()` each frame with current pitch data and target note
- Calls `renderer.update()` each frame with pitch data and exercise state
- Manages exercise state machine: idle → countdown → running → paused → complete
- Countdown: configurable 3-2-1 countdown before exercise starts (or "play when ready" mode)
- Emits events: `exercise:start`, `exercise:tick`, `exercise:note-complete`, `exercise:complete`
- Loop mode: if `config.loop === true`, restarts on complete (with brief gap)
- Duration tracking: elapsed time, active time (excluding pauses), per-note timing
- Note cursor management for sequence exercises: tracks current target note, advances based on evaluator feedback (player-driven) or timer (fixed-tempo)
- Cleans up event subscriptions on stop/destroy

## Context
This is Phase 1.3 of IMPLEMENTATION_PLAN.md. The exercise runtime is THE core abstraction — it's what makes "exercises as data" work. Every future exercise type, session runner, and renderer talks to this.

## Builder Guidance
- Certainty level: Firm — the interface is well-defined
- This replaces the tick loop from song-engine.js but is config-driven rather than hardcoded
- Must work with existing game-canvas as renderer for validation (backward compat test)
- The runtime should be agnostic to which evaluator and renderer it's using — plug-and-play

## Plan

2. State machine: idle → countdown → running → paused → complete
3. Subscribe to pitch/silence events from bus, tracked unsubs for cleanup
4. Evaluator calls: onPitch(pitchData, target), onSilence(), advanceNote()
5. Renderer calls: update(state), start(config), stop(), onCountdown(), onLoopRestart() — all optional chained for null safety
6. Four timing modes: player-driven (advance on evaluator signal), fixed-tempo (setTimeout), auto-tempo (adaptive), indefinite (no targets)
7. Note cursor management: iterate config.context.notes[], advance based on timing mode
8. Countdown: configurable seconds, emits exercise:countdown events
9. Loop: on complete if config.loop, pause for loopGapMs, reset evaluator, keep renderer, restart
10. Duration cap: stop when elapsed >= config.duration
11. Public API: start(), pause(), resume(), stop() → measurements, destroy(), getState(), getCursor(), getElapsed()
12. Events: exercise:countdown, exercise:start, exercise:tick, exercise:note-complete, exercise:complete, exercise:loop-gap, exercise:paused, exercise:resumed, exercise:stopped
